## Problem Statement

Given an array of integers `temperatures` representing daily temperatures, return an array `answer` such that `answer[i]` is the number of days you have to wait after the `i`th day to get a warmer temperature. If there is no future day with a warmer temperature, `answer[i] = 0`.

```
Input:  temperatures = [73,74,75,71,69,72,76,73]
Output: [1,1,4,2,1,1,0,0]

Input:  temperatures = [30,40,50,60]
Output: [1,1,1,0]

Input:  temperatures = [30,60,90]
Output: [1,1,0]
```

## Intuition

For each day `i`, you need the **next index `j > i` where `temperatures[j] > temperatures[i]`**. The answer is `j - i`.

Brute force scans right from every index — O(n²). The key observation: as you scan left to right, you can use a stack to remember days that are still **waiting for a warmer day**. The moment you hit a temperature that's warmer than the stack top, you've found that day's answer.

This is the **monotonic stack for next greater element** pattern.

---

## Solution: Monotonic Stack — O(n)

Maintain a stack of **indices** whose temperatures haven't found a warmer day yet (i.e., they're in a decreasing order of temperature — a monotonically decreasing stack).

```java
public int[] dailyTemperatures(int[] temperatures) {
    int n = temperatures.length;
    int[] answer = new int[n];
    Deque<Integer> stack = new ArrayDeque<>(); // indices, not temperatures

    for (int i = 0; i < n; i++) {
        // Current temperature is warmer than stack top — resolve waiting days
        while (!stack.isEmpty() && temperatures[i] > temperatures[stack.peek()]) {
            int waitingIdx = stack.pop();
            answer[waitingIdx] = i - waitingIdx;
        }
        stack.push(i);
    }

    // Remaining indices in stack never found a warmer day — answer stays 0
    return answer;
}
```

## Tracing the Example

`temperatures = [73, 74, 75, 71, 69, 72, 76, 73]`

| i | temp | stack (indices) | resolved                    |
|---|------|-----------------|-----------------------------|
| 0 | 73   | [0]             | —                           |
| 1 | 74   | [1]             | idx 0: 74>73, ans[0]=1-0=**1** |
| 2 | 75   | [2]             | idx 1: 75>74, ans[1]=2-1=**1** |
| 3 | 71   | [2,3]           | —                           |
| 4 | 69   | [2,3,4]         | —                           |
| 5 | 72   | [2,5]           | idx 4: 72>69, ans[4]=5-4=**1**; idx 3: 72>71, ans[3]=5-3=**2** |
| 6 | 76   | [6]             | idx 5: 76>72, ans[5]=6-5=**1**; idx 2: 76>75, ans[2]=6-2=**4** |
| 7 | 73   | [6,7]           | —                           |

Remaining in stack: indices 6 and 7 → `answer[6] = answer[7] = 0` (default).

Final: `[1, 1, 4, 2, 1, 1, 0, 0]` ✓

## Why Store Indices, Not Temperatures?

The answer requires the **distance** between days: `j - i`. You can't compute that from temperatures alone — you need the actual positions. Always store indices in a monotonic stack when the answer involves position arithmetic.

You can still access the temperature via `temperatures[stack.peek()]` at any time.

## Complexity

| | Time | Space |
|---|---|---|
| Brute force | O(n²) | O(1) |
| Monotonic stack | O(n) | O(n) |

Each index is pushed onto the stack exactly once and popped at most once — O(n) total operations regardless of the nested `while` loop.

## Key Insight

This is the **canonical next greater element** problem. The monotonic stack pattern:

1. Maintain a stack of "unresolved" indices in decreasing order of their values
2. When a new element is larger than the stack top, it is the "next greater" for all smaller elements currently on the stack — pop and record answers until the invariant is restored
3. Elements still in the stack after the loop never found a greater element

The decreasing order is maintained automatically — you only push after the while loop clears everything smaller. You never need to sort or explicitly enforce it.

This exact structure solves an entire family of problems:
- **Next Greater Element I & II** (#496, #503) — same pattern, circular array variant
- **Online Stock Span** (#901) — how many consecutive days ≤ current price
- **Largest Rectangle in Histogram** (#84) — next smaller element on both sides
- **Trapping Rain Water** (#42) — bounded by nearest taller walls

## Edge Cases

- **Strictly decreasing** `[90, 80, 70]` → nothing is ever popped; all answers are 0.
- **Strictly increasing** `[70, 80, 90]` → each new temperature resolves the previous day immediately.
- **All equal** `[50, 50, 50]` → no element is strictly warmer; all answers are 0. Note the `>` (strict), not `>=`.
- **Single element** `[42]` → stack never resolves; answer is `[0]`.
