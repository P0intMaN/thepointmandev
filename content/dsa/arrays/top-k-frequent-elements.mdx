## Problem Statement

Given an integer array `nums` and an integer `k`, return the `k` most frequent elements. You may return the answer in **any order**.

```
Input:  nums = [1,1,1,2,2,3], k = 2
Output: [1,2]

Input:  nums = [1], k = 1
Output: [1]
```

The problem guarantees that the answer is **unique** — there is always exactly one valid set of `k` most frequent elements.

## Step 1 — Count Frequencies

Every approach starts here: one pass with a hash map to count how many times each element appears.

```java
Map<Integer, Integer> freq = new HashMap<>();
for (int num : nums) {
    freq.merge(num, 1, Integer::sum);  // cleaner than getOrDefault
}
// [1,1,1,2,2,3] → {1:3, 2:2, 3:1}
```

The question is what you do with that frequency map next.

---

## Approach 1: Sort by Frequency — O(n log n)

Dump the map entries into a list and sort by frequency descending. Take the first `k`.

```java
public int[] topKFrequent(int[] nums, int k) {
    Map<Integer, Integer> freq = new HashMap<>();
    for (int num : nums) freq.merge(num, 1, Integer::sum);

    return freq.entrySet().stream()
        .sorted((a, b) -> b.getValue() - a.getValue())
        .limit(k)
        .mapToInt(Map.Entry::getKey)
        .toArray();
}
```

Simple but O(n log n) — we're sorting the entire frequency map just to take the top `k`.

---

## Approach 2: Min-Heap of Size k — O(n log k)

Keep a min-heap of size `k`. For every element, push it onto the heap. When the heap exceeds size `k`, pop the minimum (least frequent). Whatever remains is the top `k`.

```java
public int[] topKFrequent(int[] nums, int k) {
    Map<Integer, Integer> freq = new HashMap<>();
    for (int num : nums) freq.merge(num, 1, Integer::sum);

    // Min-heap ordered by frequency
    PriorityQueue<Integer> heap = new PriorityQueue<>(
        (a, b) -> freq.get(a) - freq.get(b)
    );

    for (int num : freq.keySet()) {
        heap.offer(num);
        if (heap.size() > k) heap.poll();  // evict least frequent
    }

    int[] result = new int[k];
    for (int i = 0; i < k; i++) result[i] = heap.poll();
    return result;
}
```

**Why a min-heap?** Counter-intuitive at first — we want the *largest*, so we use a *min* heap. The heap evicts the smallest-frequency element whenever it grows past size `k`, ensuring only the `k` largest survive.

**Complexity:** O(n log k) — each of the `n` unique elements is pushed/popped from a heap of at most size `k`.

---

## Approach 3: Bucket Sort — O(n) ✅ Optimal

The key observation: **frequency can only range from 1 to n** (an element can appear at most `n` times in an array of length `n`). So we can use the frequency as an array index.

Create `n + 1` buckets where `bucket[i]` holds all elements that appear exactly `i` times. Then scan from the back (highest frequency) and collect until we have `k` elements.

```java
public int[] topKFrequent(int[] nums, int k) {
    Map<Integer, Integer> freq = new HashMap<>();
    for (int num : nums) freq.merge(num, 1, Integer::sum);

    // bucket[i] = list of numbers that appear exactly i times
    List<Integer>[] bucket = new List[nums.length + 1];
    for (int num : freq.keySet()) {
        int f = freq.get(num);
        if (bucket[f] == null) bucket[f] = new ArrayList<>();
        bucket[f].add(num);
    }

    int[] result = new int[k];
    int idx = 0;
    // Scan from highest frequency to lowest
    for (int f = bucket.length - 1; f >= 1 && idx < k; f--) {
        if (bucket[f] != null) {
            for (int num : bucket[f]) {
                result[idx++] = num;
                if (idx == k) break;
            }
        }
    }
    return result;
}
```

No sorting, no heap — just two linear passes and an array lookup. True O(n).

## Complexity Comparison

|                | Time       | Space |
|----------------|------------|-------|
| Sort           | O(n log n) | O(n)  |
| Min-Heap       | O(n log k) | O(n)  |
| Bucket Sort    | O(n)       | O(n)  |

## Which to Use in an Interview?

Start with the **min-heap** — it demonstrates you know heaps and is the expected "smart" answer. Then, if the interviewer asks "can you do O(n)?", walk through the bucket sort. That follow-up is almost guaranteed.

## Key Insight

Bucket sort works here because the value space (frequency) is **bounded and known** — it can only be `[1, n]`. Whenever you see a problem where the range of values is bounded, think bucket sort before reaching for a heap or a sort.

This pattern reappears in:
- **Sort Colors** (Dutch flag — values bounded to {0, 1, 2})
- **Maximum Gap** (bucket the values by range)
- **Frequency Sort** (sort characters by frequency — same idea)

## Edge Cases

- **k equals the number of unique elements** — every unique element is returned; bucket sort handles this naturally.
- **All elements identical** — one element with frequency `n`, sits in `bucket[n]`, returned immediately.
- **All elements unique** — every element has frequency 1; bucket sort picks the first `k` from `bucket[1]`. Since any valid answer is acceptable, this is fine.
