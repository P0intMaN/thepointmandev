## Problem Statement

Given an array of intervals where `intervals[i] = [start_i, end_i]`, merge all overlapping intervals and return the resulting array of non-overlapping intervals.

```
Input:  intervals = [[1,3],[2,6],[8,10],[15,18]]
Output: [[1,6],[8,10],[15,18]]
Explanation: [1,3] and [2,6] overlap → merged to [1,6]

Input:  intervals = [[1,4],[4,5]]
Output: [[1,5]]
```

## Approach: Sort + Linear Merge — O(n log n)

Sort by start time. Scan left-to-right: if the current interval overlaps the last merged interval, extend it. Otherwise, start a new interval.

```java
public int[][] merge(int[][] intervals) {
    Arrays.sort(intervals, (a, b) -> a[0] - b[0]); // sort by start

    List<int[]> merged = new ArrayList<>();
    merged.add(intervals[0]);

    for (int i = 1; i < intervals.length; i++) {
        int[] last = merged.get(merged.size() - 1);
        int[] curr = intervals[i];

        if (curr[0] <= last[1]) {
            // Overlap (curr starts before or when last ends)
            last[1] = Math.max(last[1], curr[1]); // extend end
        } else {
            merged.add(curr);
        }
    }

    return merged.toArray(new int[merged.size()][]);
}
```

**Why `curr[0] <= last[1]`?** If the current interval's start ≤ last interval's end, they overlap (or touch). Touch case `[1,4],[4,5]` → `4 <= 4` → merged into `[1,5]`.

**Why `max` for the end?** Consider `[1,10],[2,3]` — the second interval is contained within the first. Without `max`, `last[1]` would shrink to 3.

## Walkthrough: `[[1,3],[2,6],[8,10],[15,18]]`

After sorting (already sorted):
- Start: merged = [[1,3]]
- [2,6]: 2 ≤ 3 → merge → last = [1, max(3,6)] = [1,6]
- [8,10]: 8 > 6 → new interval. merged = [[1,6],[8,10]]
- [15,18]: 15 > 10 → new interval. merged = [[1,6],[8,10],[15,18]] ✓

## Complexity

| | Time | Space |
|---|---|---|
| **Sort + merge** | **O(n log n)** | **O(n)** |
| Brute force (O(n²) comparisons) | O(n²) | O(n) |

Sorting dominates — the merge pass itself is O(n).

## Key Insight

After sorting by start time, the only intervals that can overlap with the current one are the immediately preceding intervals that are still "open" (haven't ended yet). Since the input is sorted by start, we only need to compare against the last merged interval.

## Edge Cases

- **Single interval**: return as-is
- **All overlapping**: one large merged interval
- **No overlapping**: all intervals returned unchanged
- **Contained intervals** `[1,10],[2,5]`: `max(10,5) = 10`, correctly kept
- **Touching** `[1,4],[4,5]`: `4 <= 4` → merged
