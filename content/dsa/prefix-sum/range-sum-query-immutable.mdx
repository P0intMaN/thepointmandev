## Problem Statement

Given an integer array `nums`, handle multiple queries of the form: return the **sum of elements between indices `left` and `right` inclusive**.

Implement the `NumArray` class:
- `NumArray(int[] nums)` — initialises the object with the array
- `int sumRange(int left, int right)` — returns the sum of `nums[left..right]`

```
Input:
["NumArray", "sumRange", "sumRange", "sumRange"]
[[[-2,0,3,-5,2,-1]], [0,2], [2,5], [0,5]]

Output: [null, 1, -1, -3]

Explanation:
sumRange(0, 2) → -2 + 0 + 3 = 1
sumRange(2, 5) → 3 + -5 + 2 + -1 = -1
sumRange(0, 5) → -2 + 0 + 3 + -5 + 2 + -1 = -3
```

The array is **immutable** — it never changes after construction.

## Why This Problem Exists

This is the foundational prefix sum problem — it exists specifically to teach you the core formula before you apply it to harder problems like Subarray Sum Equals K or 2D matrix queries.

If you compute `sumRange` naively by summing elements each time, every query costs O(k) where k is the range length. With many queries, this becomes O(n·q). Prefix sums bring every query to O(1) after an O(n) build.

## Solution

```java
class NumArray {
    private final int[] prefix;

    public NumArray(int[] nums) {
        prefix = new int[nums.length + 1];
        prefix[0] = 0; // sentinel
        for (int i = 0; i < nums.length; i++) {
            prefix[i + 1] = prefix[i] + nums[i];
        }
    }

    public int sumRange(int left, int right) {
        return prefix[right + 1] - prefix[left];
    }
}
```

That's it. `sumRange` is a single subtraction.

## How the Formula Works

`prefix[i]` stores the sum of all elements from index `0` to `i-1` (i.e. the first `i` elements):

```
prefix[0] = 0                          (empty)
prefix[1] = nums[0]
prefix[2] = nums[0] + nums[1]
prefix[i] = nums[0] + ... + nums[i-1]
```

The sum from `left` to `right` (inclusive) is:

```
sum(left, right) = prefix[right + 1] - prefix[left]
                 = (nums[0]..nums[right]) - (nums[0]..nums[left-1])
                 = nums[left]..nums[right]   ✓
```

The subtraction cancels everything before `left`, leaving exactly the range we want.

## Complexity

| Operation  | Naive  | Prefix Sum |
|------------|--------|------------|
| Build      | O(1)   | O(n)       |
| sumRange   | O(k)   | O(1)       |
| q queries  | O(n·q) | O(n + q)   |

For any non-trivial number of queries, prefix sum wins decisively.

## Key Insight

This problem is the **pure form** of the prefix sum pattern — stripped of all distractions. Once the formula `prefix[right+1] - prefix[left]` is in muscle memory, every harder prefix sum problem becomes easier to reason about.

The `+1` offset (size `n+1`, sentinel at index 0) is a deliberate design choice that avoids a special case for `left = 0`. Without it, you'd need:

```java
// Without sentinel — messier
if (left == 0) return prefix[right];
else return prefix[right] - prefix[left - 1];
```

The sentinel eliminates the branch entirely.

## Edge Cases

- **Single element** `sumRange(i, i)` → `prefix[i+1] - prefix[i] = nums[i]`. Correct.
- **Full array** `sumRange(0, n-1)` → `prefix[n] - prefix[0] = prefix[n]`. Correct.
- **Negative numbers** — prefix values may be negative or non-monotonic; the formula still holds.
- **Large values** — consider `long[]` if element values can overflow `int` across large ranges.
