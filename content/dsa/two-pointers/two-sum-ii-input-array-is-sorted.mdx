## Problem Statement

Given a **1-indexed** array of integers `numbers` that is already sorted in **non-decreasing order**, find two numbers that add up to a specific `target`. Return their indices as `[index1, index2]` where `1 <= index1 < index2 <= numbers.length`.

The solution must use **O(1) extra space**.

```
Input:  numbers = [2,7,11,15], target = 9
Output: [1,2]
Explanation: numbers[0] + numbers[1] = 2 + 7 = 9

Input:  numbers = [2,3,4], target = 6
Output: [1,3]

Input:  numbers = [-1,0], target = -1
Output: [1,2]
```

## Why the HashMap from Two Sum Won't Work Here

The original Two Sum uses a hash map to get O(n) time at the cost of O(n) space. This problem explicitly **forbids extra space** — and it doesn't need to, because there's a better property to exploit: **the array is sorted**.

Sorted order lets us make a greedy decision at every step.

---

## Approach 1: Binary Search — O(n log n)

For each element, binary search the rest of the array for its complement.

```java
public int[] twoSum(int[] numbers, int target) {
    for (int i = 0; i < numbers.length; i++) {
        int complement = target - numbers[i];
        int lo = i + 1, hi = numbers.length - 1;

        while (lo <= hi) {
            int mid = lo + (hi - lo) / 2;
            if (numbers[mid] == complement) return new int[]{i + 1, mid + 1};
            else if (numbers[mid] < complement) lo = mid + 1;
            else hi = mid - 1;
        }
    }
    throw new IllegalArgumentException("No solution");
}
```

O(n log n) — better than O(n²) brute force, but we can still do better.

---

## Approach 2: Two Pointers — O(n) time, O(1) space ✅ Optimal

Place one pointer at the start (`left`) and one at the end (`right`). Their sum falls into exactly one of three cases:

- **Sum == target** → found it, return indices
- **Sum < target** → need a bigger number, move `left` right
- **Sum > target** → need a smaller number, move `right` left

```java
public int[] twoSum(int[] numbers, int target) {
    int left = 0, right = numbers.length - 1;

    while (left < right) {
        int sum = numbers[left] + numbers[right];

        if (sum == target) {
            return new int[]{left + 1, right + 1}; // 1-indexed
        } else if (sum < target) {
            left++;
        } else {
            right--;
        }
    }

    throw new IllegalArgumentException("No solution");
}
```

Each iteration either finds the answer or eliminates one candidate — so at most `n` iterations total.

## Complexity

|               | Time       | Space |
|---------------|------------|-------|
| Brute Force   | O(n²)      | O(1)  |
| Binary Search | O(n log n) | O(1)  |
| Two Pointers  | O(n)       | O(1)  |

## Why This Works — The Core Argument

The sorted order gives us a guarantee: if `numbers[left] + numbers[right] < target`, then `numbers[left]` paired with **anything to the left of `right`** is also too small. So we can safely discard `left` as a left-side candidate and increment it.

The same logic applies in reverse: if the sum is too large, decrementing `right` is the only way to reduce it.

At every step we eliminate at least one element — the algorithm can never miss the solution while doing so. This is what makes greedy pointer movement correct here.

## Key Insight

Two pointers work when the input is **sorted** and you need to find a pair satisfying some condition. The sorted order lets you make a **guaranteed safe elimination** at every step — you don't need to check every pair, because the sort order tells you which direction to move.

This is the fundamental difference from the original Two Sum:
- **Unsorted** → use a HashMap (O(n) space)
- **Sorted** → use two pointers (O(1) space)

Recognise this trade-off. When an interviewer says "the array is sorted", immediately think: "I can probably avoid extra space here."

## Edge Cases

- **Negative numbers** `[-1, 0]` with target `-1` → works identically, no special handling needed.
- **Duplicates** `[3, 3]` with target `6` → `left=0, right=1`, sum equals target, returns `[1, 2]`. Correct.
- **Answer at the boundaries** → `[1, 2, 3]` target `4`: `left=0, right=2`, sum=4, returns `[1, 3]`. Works.
